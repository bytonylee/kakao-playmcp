# 민원런·내꺼리콜 PlayMCP 설계

작성일: 2026-07-14  
신청자: 이정민  
소속: 개인 / 개인개발·프리랜서  
홈페이지: https://tonylee.im  
GitHub: https://github.com/bytonylee

## 목표와 포지셔닝

Agentic Player 10에 서로 다른 생활 문제를 해결하는 MCP 서버 두 개를 출품한다.

- 민원런: 온라인 처리 가능 여부, 민원실 운영시간, 실시간 대기 인원, 준비물을 비교해 헛걸음을 줄인다.
- 내꺼리콜: 제품 정보와 공식 인증·리콜 정보를 대조해 대상 가능성과 다음 행동을 안내한다.

민원런은 반복되는 시간 낭비를 줄이는 실용 서비스, 내꺼리콜은 비용과 안전 피해를 예방하는 기억에 남는 서비스로 포지셔닝한다. 두 서비스는 한 Git 저장소에서 관리하지만 코드, Dockerfile, Endpoint, PlayMCP 등록과 심사를 독립적으로 운영한다.

## 민원런

대표 질문: "강남역 근처에서 등본을 떼려는데 지금 어디로 가야 가장 빠를까?"

처리 순서:

1. 온라인 처리 가능 여부를 먼저 확인한다.
2. 위치와 업무 종류를 기준으로 민원실을 찾는다.
3. 운영시간, 실시간 대기 인원, 거리를 비교한다.
4. 추천 방문처와 대안, 준비물, 수수료를 제시한다.
5. 조회 시각과 공식 출처를 표시한다.

초기 지원 민원은 주민등록표 등·초본, 인감증명, 전입신고, 여권, 가족관계증명서다.

도구:

- `plan_civil_visit`: 위치와 민원 종류를 받아 전체 방문 계획 생성
- `compare_civil_offices`: 주변 민원실의 거리, 운영시간, 대기 정보 비교
- `get_civil_checklist`: 온라인 처리 여부, 준비물, 수수료 안내

민원 예약·결제, 주민번호가 필요한 개인 업무, 정확한 대기시간 보장은 범위에서 제외한다.

## 내꺼리콜

대표 질문: "중고로 산 A123 카시트가 리콜 대상인지 확인해줘."

처리 순서:

1. 제품명, 모델명, 제조사, 제조 시기, 인증번호를 정규화한다.
2. 국내·외 공식 인증 및 리콜 후보를 검색한다.
3. 모델명, 생산 기간, 인증번호를 대조한다.
4. `대상 일치`, `추가 확인 필요`, `일치 정보 없음`으로 구분한다.
5. 위해 사유와 사용 중지, 수리, 교환, 환불 절차를 안내한다.
6. 제조사 문의에 필요한 사진과 제품 정보를 정리한다.

초기에는 전기용품, 생활용품, 어린이제품과 국내·외 제품안전 리콜을 지원한다. 본선에서는 식품, 의약품, 자동차 리콜과 가족용품 일괄 점검을 확장한다.

도구:

- `search_product_safety`: 인증·리콜 후보 검색
- `verify_recall_match`: 모델, 제조 시기, 인증번호의 일치 수준 판정
- `make_recall_action_plan`: 사용 중지와 교환·환불·수리 절차 생성
- `check_products_batch`: 여러 제품 일괄 점검

제품 안전 보장, 근거 없는 임의 판정, 제조사 대신 접수하는 기능은 범위에서 제외한다.

## 저장소와 배포 구조

```text
kakao-playmcp/
├── minwon-run/
│   ├── src/
│   ├── test/
│   ├── Dockerfile
│   └── package.json
├── naekkeo-recall/
│   ├── src/
│   ├── test/
│   ├── Dockerfile
│   └── package.json
└── docs/
```

- 각 서비스는 별도 프로세스와 컨테이너로 빌드한다.
- 초기에는 실행 시 공유 패키지를 두지 않고 배포 독립성을 우선한다.
- 저장소는 공개로 운영해 PlayMCP in KC에 GitHub PAT를 전달하지 않는다.

## 기술 구조와 개발가이드 준수

- TypeScript, Node.js 22 LTS, 공식 MCP TypeScript SDK
- Streamable HTTP, 공개 Remote MCP, Stateless, no session
- 기본 포트 `8000`, MCP 주소 `/mcp`, 상태 확인 `/healthz`
- 인증 없음, 영구 데이터 저장 없음, 짧은 메모리 캐시만 사용
- MCP 지원 버전 `2025-03-26`부터 `2025-11-25` 사이에서 협상
- 서버와 도구 이름에 `kakao` 사용 금지
- 서버별 도구 3~10개 유지
- 모든 도구에 name, description, inputSchema와 5개 annotations 지정
- 원본 API 대신 필요한 정보만 정제해 반환
- 캐시 적중 평균 100ms, 전체 p99 3,000ms 목표

데이터 흐름은 `PlayMCP 질문 → 입력 검증 → 질의 정규화 → 캐시 → 공식 API → 응답 검증 → 비교·판정 → 근거·조회시각·행동 반환`이다.

## 데이터와 판정

민원런은 행정안전부 민원실 실시간 대기현황 API와 검증된 민원 준비물 자료를 사용한다. 1차 위치 검색은 행정구역·기관명으로 제공하고 카카오 로컬 API 키 확보 후 거리순 검색을 추가한다.

내꺼리콜은 국가기술표준원 제품 안전인증 및 리콜 API를 사용한다. 제품명, 모델명, 제조사, 인증번호, 제조 기간을 내부 공통 형식으로 정규화한다.

공통 오류 처리:

- 외부 API 제한시간 2초, 전체 도구 제한시간 2.8초
- 시간 초과, 비정상 응답, 필수 필드 누락은 사용자용 오류로 변환
- 이전 캐시 사용 시 조회 시각을 표시
- 근거가 없으면 추측하지 않음
- 원본 API 내부 코드와 전체 응답을 노출하지 않음

내꺼리콜의 `일치 정보 없음`은 안전 보장이 아니라 조회된 공식 데이터에서 일치 항목을 찾지 못했다는 뜻으로만 표현한다.

## 보안 전략

- API 인증정보, GitHub 토큰, 쿠키와 비밀번호를 저장소와 대화에 넣지 않는다.
- 로컬 비밀 설정과 인증서 파일을 Git 추적 및 Docker 빌드 대상에서 제외한다.
- 공개 Git 저장소를 사용하고 KC의 PAT 입력은 비워둔다.
- 운영 API 인증정보는 런타임 Secret 입력 기능이 확인된 경우에만 주입한다.
- KC가 Secret 주입을 지원하지 않으면 인증정보를 이미지에 포함하지 않는다. 키 없는 공식 데이터나 별도 비밀 프록시를 검토하고, 안전하지 않으면 배포를 중단한다.
- 외부 요청 도메인을 공식 API로 고정해 SSRF를 차단한다.
- 입력 길이, 형식, 응답 크기, 동시 요청과 호출 빈도를 제한한다.
- 검색어 원문과 개인식별정보를 저장하거나 기록하지 않는다.
- 로그에는 요청 ID, 도구명, 처리시간, 결과 건수와 오류 종류만 기록한다.
- 다단계 컨테이너 빌드, 고정 lockfile, 비root 사용자, `linux/amd64` 검증을 적용한다.
- 커밋 전 비밀정보 탐지, 의존성 감사와 컨테이너 취약점 검사를 실행한다.

## 검증 기준

- 도구 스키마와 annotations 자동 검증
- 정상, 빈 결과, 오류, 시간 초과 API fixture 테스트
- 민원실 운영 종료와 대기값 없음 경계 테스트
- 리콜 완전 일치, 부분 일치, 불일치, 복수 후보 테스트
- 위험한 입력과 과도한 길이 처리 테스트
- MCP Inspector의 initialize, tools/list, tools/call 성공
- Docker 로컬 실행, `/healthz`, `linux/amd64` 확인
- PlayMCP의 `정보 불러오기` 성공
- AI 채팅에서 서비스별 대표 대화 3개와 오류 대화 성공

일반 카카오클라우드가 아니라 공모전 전용 `https://playmcp.kakaocloud.io`에서 Git 소스 빌드로 생성한다.

| 입력값 | 민원런 | 내꺼리콜 |
| --- | --- | --- |
| MCP 서버 이름 | `minwon-run` | `naekkeo-recall` |

| 브랜치/ref | `main` | `main` |

## 실행 순서

1. 저장소와 두 서비스의 테스트 골격을 만든다.
2. 민원런 데이터 어댑터와 세 도구를 구현한다.
3. 내꺼리콜 검색·대조 엔진과 네 도구를 구현한다.
4. 오류 처리, 캐시, 보안 로그와 제한시간을 적용한다.
5. 자동 테스트와 MCP Inspector 검증을 통과한다.
6. 두 Docker 이미지를 `linux/amd64`로 빌드·검증한다.
7. 공개 GitHub 저장소에 push한다.
8. PlayMCP in KC에 서버 두 개를 생성한다.
9. PlayMCP에 각각 임시 등록하고 AI 채팅으로 검증한다.
10. 심사 요청, 승인, 전체 공개를 완료한다.
11. 공개 URL을 비즈니스폼에 입력하고 최종 제출한다.

## 현재 차단 요소

- `playmcp.kakaocloud.io`는 별도 로그인 화면이므로 사용자의 카카오 로그인이 한 번 필요하다.
- 공공 API 인증정보 발급과 KC의 안전한 Secret 주입 지원 여부를 배포 전에 확인해야 한다.
- 예선 접수 마감일이 2026-07-14이므로 심사 승인과 전체 공개가 마감 전에 완료되지 않을 위험이 매우 높다.
